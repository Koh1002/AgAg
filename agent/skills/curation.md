# スキル: キュレーション(有用度評価)

ネタを選ぶときの評価基準。経験に基づいて随時更新すること。

## スコアリング(1〜5)

| 観点 | 高評価 | 低評価 |
|---|---|---|
| 実用性 | 今日から使えるツール・手法・コード | 抽象論だけ、実装なし |
| 新規性 | 新リリース・新手法・新知見 | 既知の話の焼き直し、まとめ記事のまとめ |
| 信頼性 | 一次ソース(公式ブログ・論文・リポジトリ) | 伝聞、誇張されたスレッド |
| 反響 | HN/Reddit で高スコア・活発な議論 | 反応ゼロの宣伝投稿 |
| 成長寄与 | 自分(AgAg)の改善に直接使える | 自分の仕組みと無関係 |

## 除外ルール

- 単なる製品宣伝・アフィリエイト・ウェイトリスト募集
- タイトル釣り(内容が伴わない「○○が革命的」系)
- 同一ネタの重複(最も一次に近いソースを1件だけ残す)
- 2週間以上前の話題(ただし「今日再注目された」文脈があれば可)
- GitHubの「SEOキーワード詰め込み型」量産リポジトリ: 「Top ◯◯ Tools/Simulator for LLM 20XX」のような没個性なタイトルが、近い投稿時刻で複数アカウントから並んで出てくる場合はスパム的な検索流入狙いを疑う。`ai-agent`等の監視キーワードにヒットするためだけに作られ、README以上の実体がないことが多い(2026-07-07の収集データで複数件確認)
- 製品発表・料金変更・仕様変更などの「一次情報がはっきり存在するはずの」ニュースをWeb補完調査する場合、検索結果は同一内容を微妙な言い回しだけ変えて再掲するSEOニュースアグリゲーターサイトで埋まりやすい(2026-07-08、Fable 5の課金体系変更で確認)。運営元(Anthropic等)の公式ブログ・ニュースページをWebFetchで直接確認し、そちらをURLとして採用する。二次サイトの記述だけを鵜呑みにしない
- **候補を`items`に入れる前に、必ずタイトル/URLで過去の`data/digests/*.json`を検索して既出でないか確認する**([[growth-scan.md]]の2026-07-12記録「GitHub/RSSの再浮上」参照)。GitHubのスター数上位リポジトリは公開から2週間近く同じ検索結果に居座り続けるため、スコアが高いというだけで採用すると同一リポジトリを複数日連続で掲載してしまう(2026-07-18・2026-07-20と2日連続で`xai-org/grok-build`を掲載していたことを2026-07-22に確認・以降除外)。既出であれば「同じ事象の重大な進展」でない限りその日のitemsからは外す

## カテゴリ判定の目安

- **新モデル・新製品**: モデル・API・製品のリリース/アップデート
- **フレームワーク・ツール**: エージェント構築のためのOSS・SDK・プロトコル(MCP等)
- **研究**: 論文・ベンチマーク・評価手法
- **事例・ノウハウ**: 実装記録・チュートリアル・運用知見
- **議論・意見**: 業界動向・設計思想・批評

## バランス

- 1日のダイジェストにカテゴリが偏りすぎないようにする(全部論文、全部宣伝は避ける)
- 日本語ソース(Zenn/Qiita)から良質なものがあれば1〜2件は含める(読者は日本語話者)
- 自分の成長に使えるネタ(エージェント設計・自己改善手法・評価手法)は score +1 の重み付け
