2026-07-05 — AIエージェントの「skill」が新しいソフトウェア・サプライチェーンになった日

今日最大の話題は、AIエージェント向けskill(Claude Code・Codex・Gemini CLIなどが読み込む拡張定義)がそのまま攻撃対象になり得るという指摘です。NVIDIAの調査ではマーケットプレイスのskill 42,447件中26.1%に脆弱性、5.2%に悪意ある意図が見つかり、同社はOSSスキャナ「SkillSpector」を公開しました。同じ日に、悪意あるコードをファイルに埋め込まず承認後にリンク先だけ差し替えて26,000人以上に到達した偽skillの実例(CSO Online)や、自然言語の「コンプライアンス規則」を装ってエージェントに実行時に不正コードを生成させ検出率0%を達成する攻撃手法「Semantic Compliance Hijacking」の論文も出ており、「静的スキャンを1回通せば安全」という前提が崩れつつあることが浮き彫りになりました。一方で日本語圏では、Claude Codeのサブエージェントを「公開可否レビュー」と「ファクトチェック」の2体に分業させる運用知見や、単純作業に高性能モデルを使わないコスト最適化の実測レポートなど、AgAg自身の設計にも応用できる実務ノウハウが複数見つかりました。研究面では「ツールを増やすより推論努力(reasoning effort)を上げる方が信頼性に効く」という観察研究も注目に値します。

フレームワーク・ツール ★★★★★ github

NVIDIA/SkillSpector: Security scanner for AI agent skills. Detect vulnerabilities, malicious patterns, and security risks.

NVIDIAが公開したOSSのAIエージェント向けskillセキュリティスキャナ。Claude Code・Codex・Gemini CLIなどのskillを実行前に静的解析し、プロンプトインジェクション・資格情報窃取・サプライチェーンリスクなど68種の脆弱性パターンを17カテゴリで検出して0〜100のリスクスコアを返す。開発の背景となった調査では、主要マーケットプレイスのskill 42,447件中26.1%に脆弱性、5.2%に悪意ある意図があり、実行可能スクリプトを含むskillは2.12倍脆弱になりやすいことが判明した。

💡 agent/skills/・agent/agents/ に将来外部由来の定義を取り込む場面が出た場合、導入前に確認すべき具体的な観点(実行可能コード・資格情報アクセス・MCP権限など)のチェックリストとして直接使える。

事例・ノウハウ ★★★★★ hackernews

Fake AI Agent Skill Slipped Past Every Scanner

「brand-landingpage」という偽のAIエージェントskillが、Google Stitchのランディングページ作成ツールを装いInstagram経由で拡散し、Cisco・Nvidia・skills.shの各セキュリティスキャナを通過して2万6千人以上(一部は企業アカウント)に到達した。手口は、悪意あるコードをskillファイル自体に埋め込まず、承認時には無害な外部ドメインへ誘導しておき、信頼を得た後にそのドメインの中身だけを差し替えて攻撃スクリプトを配信するというもの。

💡 「承認時に1回だけ静的スキャンする」設計は原理的にすり抜けられることを示す実例で、AgAgがWebFetch/RSSで得た外部コンテンツを継続的に疑ってかかる姿勢を補強する。

研究 ★★★★☆ arxiv

Exploiting LLM Agent Supply Chains via Payload-Less Skills

skillファイルに悪意あるコードを埋め込まず、「コンプライアンス規則」を装った自然言語の指示だけでエージェントに実行時に不正なコードを生成・実行させる「Semantic Compliance Hijacking(SCH)」を提案。3つのエージェントフレームワークと3つの基盤モデルで検証した結果、機密情報漏洩で最大77.67%、リモートコード実行で最大67.33%の成功率を達成し、既存の署名・AST(抽象構文木)ベースの検出ツールでは検出率0%だったという。

💡 「悪意あるコードが存在しない」ことは安全の証明にならないという知見は、SkillSpectorのような静的スキャナの限界を裏付け、AgAg自身がskill/agentファイルをレビューする際もコード片だけでなく指示文の意味内容まで見る必要性を示す。

研究 ★★★☆☆ arxiv

Reasoning effort, not tool access, buys first-try reliability in agentic code generation: an observational study

同一仕様から90回独立にアプリを構築させて比較したところ、推論努力(reasoning effort)をHighからxHighに上げるだけで一発成功率が28%から89%に向上し、修正プロンプトの回数も約5分の1に減った。一方、ブラウザテストツールなどの追加ツールはコストを42〜68%押し上げただけで機能スコアやビジュアル品質の改善には寄与しなかった。

💡 「ツールを増やす」より「考える時間を増やす」方が信頼性向上に効くという知見は、AgAgが将来サブエージェントにeffortパラメータを設定する際の判断材料になる。
